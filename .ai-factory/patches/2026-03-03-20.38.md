# Stabilize postgres runtime and schema source

**Date:** 2026-03-03 20:38
**Files:** compose.production.yml, .ai-factory/DESCRIPTION.md, db/migrations/versions/20260303_000001_create_leads_tables.py, db/migrations/.gitkeep
**Severity:** medium

## Problem

Post-migration review found two operational risks:
- Production postgres service used forced `user: "999:999"`, which can break official `postgres` image initialization and data directory permissions.
- Schema had two concurrent sources of truth (`db/schema.sql` and Alembic migration file), increasing drift risk.

## Root Cause

Security hardening and migration scaffolding were introduced quickly in one large change set:
- `user` override was copied as a generic hardening pattern without accounting for postgres image startup assumptions.
- Alembic-like migration artifact was added without full Alembic runtime integration, while bootstrap SQL was already active in compose.

## Solution

- Removed forced `user` override from postgres in production compose so container runs with image defaults.
- Removed orphan migration file and kept bootstrap schema as canonical source.
- Added `db/migrations/.gitkeep` to preserve directory intent without implying executable migrations.
- Updated project description to state bootstrap DDL via `db/schema.sql`.

## Prevention

- Do not apply non-root user hardening blindly to stateful vendor images; verify image entrypoint requirements first.
- Keep exactly one schema authority per iteration unless migration framework is fully wired (config, dependency, execution path).
- During review, add explicit checklist item: "stateful container startup compatibility" and "schema source-of-truth".
