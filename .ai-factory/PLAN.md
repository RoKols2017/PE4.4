## Commit Plan
- **Commit 1** (after tasks 1-3): `refactor(domain): unify validators and guard semantics across channels`
- **Commit 2** (after tasks 4-7): `refactor(dialog): align response policy and confirm edit flow`
- **Commit 3** (after tasks 8-9): `test(docs): align bot and web workflow coverage`

## Tasks

### Phase 1: Shared validation and guard semantics
- [ ] Task 1: Unify `validate_name` and `normalize_name` behavior between `bot/domain.py` and `web_assistant/domain.py`. Prefer a shared pure-Python helper module (or one canonical implementation mirrored with strict parity) for intro-prefix stripping, contact rejection, request-like phrase rejection, allowed characters, minimum length, and the exact validation-code set (`name_looks_like_contact`, `name_looks_like_request`, `name_invalid_chars`, etc.) so both channels accept and reject the same name inputs for the same reasons. Logging: DEBUG normalized/raw name comparison and matched guard reason, INFO accepted normalized names, WARNING rejected names with validation code, ERROR unexpected parsing failures.
- [ ] Task 2: Unify request-like and contact-like guards in `bot/domain.py` and `web_assistant/domain.py` for all three steps (`name`, `contact`, `request`). Keep semantics and validation codes aligned across channels while allowing channel-specific parsing adapters for contact types (`telegram_username` vs `email`). Ensure the shared guard layer defines what counts as request-like/contact-like, and channel adapters only decide how valid contact values are parsed. Logging: DEBUG field classifier inputs/results, INFO accepted field type, WARNING guard-triggered retry with validator code, ERROR contradictory parser outcomes.
- [ ] Task 3: Review and align deterministic retry messages and fallback reasons in `bot/runtime_policy.py` and `web_assistant/runtime_policy.py` after Task 1 and Task 2 stabilize shared validator codes. The same validation codes should produce equivalent user-facing explanations, differing only where channel-specific contact examples are required. Logging: DEBUG selected retry template and fallback reason, INFO fallback generation success, WARNING missing template/code mapping, ERROR formatter/schema mismatches.

### Phase 2: Response policy and FSM parity
- [ ] Task 4: Unify `response_policy` contract and allowed intents between `bot/runtime_policy.py`, `web_assistant/runtime_policy.py`, `bot/ai_logic.py`, and `web_assistant/ai_logic.py`. Prefer one shared source of truth for intent constants, schema keys, and parser rules so both channels share the same JSON schema, intent whitelist, parser behavior, and deterministic fallback triggers. Logging: DEBUG prompt contract version and allowed intents, INFO parsed policy result, WARNING unknown/disallowed intent fallback, ERROR malformed model payload.
- [ ] Task 5: Align mixed-input resolver semantics in `bot/bot.py` and `web_assistant/routes.py`. Both channels should only cascade across multiple fields when intent is explicitly `mixed_input`; otherwise they must mutate and validate only the current step. Preserve channel-specific contact parsing while keeping step progression identical, and ensure existing valid fields are not revalidated or cleared unless the edited/current field requires it. Logging: DEBUG resolver mode (`single-step` vs `mixed-input`) and accepted candidate fields, INFO deterministic step progression, WARNING invalid candidate blocked by validator, ERROR illegal cascade or skipped-step condition.
- [ ] Task 6: Align confirm-edit behavior in `bot/bot.py`, `web_assistant/routes.py`, `bot/session.py`, and `web_assistant/session.py`. Editing one field from confirm must return only to that field, preserve the other valid fields, and re-enter confirm without unexpectedly re-running unrelated steps. Explicitly align the hidden session metadata contract (`confirm_edit_target`, `last_validation_result`, `accepted_candidate_fields`, etc.) and reset/save semantics between bot and web so both channels carry the same confirm-edit assumptions. Logging: DEBUG selected edit target and preserved draft fields, INFO successful re-entry to confirm, WARNING ambiguous edit request routed to clarification, ERROR inconsistent confirm-edit state.
- [ ] Task 7: Add or extract a shared conversation utility layer for parity-critical pieces that should not drift again: validator-code constants, request/contact guard helpers, retry/fallback code maps, and any intent/schema constants that bot and web both consume. Place it in a shared pure-Python module if feasible; otherwise codify strict mirrored ownership and parity tests. Logging: DEBUG shared utility inputs/outputs, INFO parity-critical helper usage, WARNING channel divergence detected in tests or runtime assertions, ERROR unsupported shared-code assumptions.

### Phase 3: Regression coverage and documentation parity
- [ ] Task 8: Create matching regression scenarios for bot and web in `bot/tests/` and `web_assistant/tests/`: request-like text on name step, contact-like text on request step, explicit mixed-input success, confirm-stage field edits, and “do not cascade after edit unless mixed_input intent” behavior. Keep test names, ordering, and expectations as parallel as possible across channels, ideally as one mirrored parity matrix so future drift is obvious. Logging in tests: assert validation codes, preserved fields, deterministic next-step outcomes, and symmetric fallback reasons where semantics are shared.
- [ ] Task 9: Update `docs/workflow.md`, `docs/ru/workflow.md`, `docs/testing.md`, `docs/ru/testing.md`, `docs/api.md`, `docs/ru/api.md`, `README.md`, and `.ai-factory/ARCHITECTURE.md` to document which behaviors are now intentionally unified across bot/web and which remain channel-specific (`telegram_username` vs `email`, transport, and session identifiers). Explicitly state that only contact parsing, transport, and identity source differ, while validator semantics, retry codes, response_policy contract, and confirm-edit behavior are shared. Logging documentation: note the shared validator codes and fallback reasons that should now appear symmetrically in both channels.
