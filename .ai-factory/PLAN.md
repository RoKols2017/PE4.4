# Implementation Plan: Deterministic response_policy over FSM for human-like lead capture

Branch: none (fast mode)
Created: 2026-03-22

## Settings
- Testing: yes
- Logging: verbose
- Docs: yes

## Commit Plan
- **Commit 1** (after tasks 1-3): `feat(dialog): add structured runtime contract and response policy`
- **Commit 2** (after tasks 4-7): `feat(flow): support mixed-input parsing and partial field edits`
- **Commit 3** (after tasks 8-10): `test(docs): cover deterministic fallback and document policy flow`

## Tasks

### Phase 1: Narrow AI contract and shared response policy
- [x] Task 1: Introduce a shared structured runtime contract for `gpt-4o-mini` in `bot/ai_logic.py`, `web_assistant/ai_logic.py`, and a shared pure-Python helper module for schema constants/parsing (or mirrored modules with one contract source). Define the exact JSON schema, allowed intents, schema parsing/validation, and deterministic fallback behavior when the model returns empty text, invalid JSON, unknown intent, malformed candidate fields, or unexpected top-level keys. Ensure the same contract is consumed by AI wrappers, response_policy, and tests so bot/web do not drift. Logging: DEBUG prompt inputs and parsed model payload keys without PII leakage, INFO successful structured response parsing, WARNING fallback activation and invalid intent/schema cases, ERROR OpenAI/API or schema processing failures with correlation fields (`step`, `session_id`/`chat_id`).
- [x] Task 2: Redesign prompts for `gpt-4o-mini` to be short, low-ambiguity, and schema-first in `bot/ai_logic.py` and `web_assistant/ai_logic.py`. Encode a narrow list of allowed intents (for example `answer_current_step`, `edit_name`, `edit_contact`, `edit_request`, `mixed_input`, `clarify`, `confirm_yes`, `confirm_no`, `unknown`) and explicitly forbid state control by the model. Ensure all runtime outputs are JSON-only, confirm/edit intent resolution uses the same contract, and the deterministic fallback path is always available. Logging: DEBUG selected prompt template and allowed intents set, INFO model call success per step, WARNING model output normalized/trimmed to fallback-safe values, ERROR unexpected API response structure.
- [x] Task 3: Replace free-form AI reply usage with a `response_policy` layer in `bot/bot.py` and `web_assistant/routes.py`. The layer must accept `current_step`, `known_fields`, `validation_result`, `offscript_count`, and `last_user_message`, and return only the minimal structured policy result (`detected_intent`, `candidate_fields`, `needs_clarification`, `user_facing_message`). A separate deterministic resolver/applicator in code must remain responsible for draft mutation, field acceptance, next-step choice, and persistence so the FSM stays the single source of truth. Logging: DEBUG response_policy inputs/outputs and chosen deterministic branch, INFO step transitions and successful policy decisions, WARNING policy/model disagreement resolved in favor of validator, ERROR inconsistent policy output that requires hard fallback.

### Phase 2: Deterministic extraction, edits, and retry UX
- [x] Task 4: Add mixed-input parsing support in `bot/domain.py`, `web_assistant/domain.py`, and the deterministic resolver/policy orchestration modules. The model may propose candidate values for `name`, `contact`, and `request` from one free-form message, but the backend must validate each candidate independently before mutating draft state. Support channel-specific contact semantics (`phone|telegram_username` for bot, `phone|email` for web), partial acceptance of only valid fields, and deterministic prompting for only the missing/invalid field. Logging: DEBUG candidate field extraction and field-by-field validator results, INFO partial draft enrichment from one message, WARNING rejected candidate field with validator code, ERROR contradictory candidate parse that cannot be reconciled.
- [x] Task 5: Extend in-memory session models in `bot/session.py` and `web_assistant/session.py` to track metadata needed by the new deterministic policy flow before confirm-edit logic is introduced (for example `last_user_message`, `last_validation_result`, `confirm_edit_target`, `accepted_candidate_fields`, or mixed-input traces). Update reset/save helpers so partial edits, mixed-input acceptance, and deterministic fallbacks remain consistent after each message. Logging: DEBUG session mutations and stored policy metadata, INFO session resets and confirm-edit lifecycle events, WARNING suspicious stale state or repeated loop detection, ERROR serialization/state-shape mismatches.
- [x] Task 6: Replace `confirm -> да/нет` with field-level editing in `bot/bot.py`, `web_assistant/routes.py`, and the deterministic resolver/session helpers. Support user intents like “исправь контакт”, “измени имя”, “поменяй задачу”, return only to the requested field, preserve other valid draft data, and come back to confirm after the edited field is revalidated. Logging: DEBUG detected edit intent and chosen field rollback, INFO field-specific return to step and successful revalidation, WARNING unsupported edit intent routed to clarification, ERROR illegal state transition from confirm/edit flow.
- [x] Task 7: Rework retry UX and deterministic user-facing fallbacks in `bot/bot.py`, `web_assistant/routes.py`, and prompt helpers so error messages explain the problem type in plain language instead of only rejecting input. Keep replies short, natural, non-marketing, and context-sensitive, while preserving backend control. Ensure that if the model message is empty, malformed, or weak, the system still returns a high-quality deterministic retry prompt. Logging: DEBUG selected retry template and validation reason, INFO retry response generated successfully, WARNING repeated retries / high `offscript_count`, ERROR formatting failures in fallback/retry composition.

### Phase 3: Session/state adjustments and regression coverage
- [x] Task 8: Add targeted tests for the new contract and failure modes in `bot/tests/test_domain.py`, `bot/tests/test_dialog.py`, `web_assistant/tests/test_domain.py`, `web_assistant/tests/test_routes.py`, plus dedicated AI/policy unit tests for schema parser and deterministic resolver if needed. Cover malformed JSON from the model, unknown intents, empty model output, unexpected keys, mixed input containing valid+invalid fields, partial edits from confirm, validator-vs-model disagreement, and deterministic fallback behavior. Explicitly cover the current asymmetric risk that web already uses `FakeAI` route tests while Telegram lacks equivalent dialog-level AI boundary tests. Logging expectations in tests: assert backend state/step control remains deterministic and no invalid candidate is committed.
- [x] Task 9: Update docs in `README.md`, `docs/workflow.md`, `docs/api.md`, `docs/architecture.md`, `docs/testing.md`, `docs/ru/workflow.md`, relevant Russian mirrors, and `AGENTS.md` to describe the new response_policy layer, structured AI contract, confirm edit flow, mixed-input parsing, and the rule that validators override model suggestions. Logging documentation: capture recommended correlation fields and fallback/debug signals for model-contract failures.
- [x] Task 10: Add a focused implementation note to the plan/docs or architecture artifacts (as appropriate during implementation) that codifies the separation between `response_policy`, deterministic resolver, and persistence boundary so future changes do not reintroduce free-form model control over state transitions or commit decisions. Files: likely `.ai-factory/ARCHITECTURE.md`, `docs/architecture.md`, and `docs/workflow.md`. Logging: document where fallback and validator-overrides should be observed in runtime logs.
