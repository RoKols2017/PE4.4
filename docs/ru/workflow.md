# Workflow сбора заявок

Английская версия: [Workflow](../workflow.md)

## Обзор

Оба канала используют один и тот же pipeline сбора заявки:

1. собрать `name`
2. собрать `contact`
3. собрать `request`
4. запросить `confirm`
5. сохранить лид в PostgreSQL и сбросить сессию

Основное различие только в транспортном слое:

- Сайт использует HTTP endpoint'ы и браузерный виджет.
- Telegram использует команды бота и текстовые сообщения.

## Общие правила flow

- На каждом шаге выполняется валидация до перехода дальше.
- Если значение похоже на другой тип поля, пользователь остается на текущем шаге.
- Невалидные попытки накапливаются в `offscript_count` и `qa_flags`.
- AI ограничен узким JSON-контрактом: intent-подсказка, candidate-поля, флаг уточнения и короткая пользовательская реплика.
- AI используется только для phrasing, intent hints и candidate extraction; он не управляет переходами FSM и не решает, что сохранять.
- Если ответ модели пустой, битый или противоречит валидатору, backend обязан перейти на deterministic fallback.
- После финального подтверждения лид сохраняется в PostgreSQL, а сессия очищается.

## Workflow на сайте

### Точка входа

- Пользователь открывает `/` и работает со встроенным виджетом.
- Frontend начинает сессию через `POST /api/chat/start`.
- Backend создает или переиспользует `session_id` и возвращает первый prompt для `name`.

### Пошаговый flow

1. `name`
   - Ввод проверяется через `validate_name(...)`.
   - Валидные имена нормализуются через `normalize_name(...)`.
   - При ошибке пользователь остается на том же шаге.
2. `contact`
   - Ввод разбирается как `phone` или `email`.
   - Валидация отклоняет текст запроса и невалидные контакты.
3. `request`
   - Ввод проверяется как короткое описание задачи.
   - Контактоподобные значения отклоняются и запрашиваются заново.
4. `confirm`
   - Пользователю показывается summary с `name`, `contact` и `request`.
   - `да` сохраняет лид.
   - `нет` сбрасывает flow обратно на `name`.
   - `исправь имя` / `исправь контакт` / `исправь задачу` возвращает только к выбранному полю.

### Сохранение

- При успешном подтверждении вызывается `save_website_lead(...)`.
- Запись сохраняется с `lead_id`, `session_id`, источником и quality metadata.
- После сохранения in-memory сессия сайта сбрасывается.

## Workflow в Telegram

### Точка входа

- Пользователь начинает с `/start` или перезапускает flow через `/new`.
- Бот сбрасывает сессию, привязанную к `chat_id`.
- Затем бот отправляет вводное сообщение и запрашивает `name`.

### Пошаговый flow

1. `name`
   - Ввод проходит валидацию и нормализацию.
   - При ошибке пользователь остается на текущем шаге.
2. `contact`
   - Ввод разбирается как `phone` или `telegram_username`.
   - Username из профиля Telegram не подставляется скрыто как fallback-контакт; бот просит явный контакт от пользователя.
3. `request`
   - Ввод должен описывать задачу пользователя.
   - Контактоподобный текст отклоняется.
4. `confirm`
   - Бот показывает summary с `name`, `phone`, `telegram` и `request`.
   - `да` сохраняет лид.
   - `нет` сбрасывает flow и запускает его заново.
   - `исправь имя` / `исправь контакт` / `исправь задачу` возвращает только к выбранному полю.

## Слой Response Policy

- `gpt-4o-mini` возвращает только JSON.
- Схема ответа ограничена полями `detected_intent`, `candidate_fields`, `needs_clarification`, `user_facing_message`.
- Candidate-поля из mixed input принимаются только после детерминированной валидации каждого поля.
- Если модель предлагает невалидное значение, источник истины - backend-валидатор.
- Confirm, field edits, retries и решение о сохранении выполняются deterministic-кодом, а не моделью.

### Сохранение

- При успешном подтверждении создается lead record и вызывается `save_lead(...)`.
- Запись сохраняется с `lead_id`, метаданными источника пользователя и quality metadata.
- После сохранения Telegram-сессия сбрасывается.

## Текущие слабые места

- Состояние сессий хранится только в памяти, поэтому незавершенные диалоги теряются после рестарта.
- Узкий контракт с моделью требует строгих тестов, иначе ошибки intent/schema быстро ломают UX.
- Перед сохранением нет встроенной дедупликации и антиспам-защиты.

## Рекомендации по улучшению

### Надежность

- Вынести session state из памяти в PostgreSQL или Redis.
- Добавить TTL и очистку заброшенных сессий.
- Добавить поиск дублей по контакту и недавним заявкам.

### UX

- Показывать явный прогресс вида `Шаг 1 из 4`.
- Разрешить редактирование `name`, `contact` и `request` прямо на этапе подтверждения.
- Возвращать более точные retry-сообщения, например: “это похоже на запрос, а здесь нужен контакт”.

### Конверсия

- Поддержать несколько контактных значений в одном сообщении там, где это возможно.
- Использовать известный website context для prefill контакта или пропуска части шагов.
- Уменьшить число полных перезапусков, если ошибка только в одном поле.

### Аналитика

- Считать, на каком шаге пользователи чаще всего отваливаются.
- Разделять recovery-метрики для сайта и Telegram.
- Сохранять структурированные причины retry и restart для анализа funnel.
